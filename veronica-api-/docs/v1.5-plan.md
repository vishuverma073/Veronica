## 2. New v1.5 features

### 2.1 Returns / RMA
- Schema: `return_requests` (order_id, items, reason, status, refund_amount), state machine `requested → approved → picked_up → received → refunded | rejected`.
- Customer: request a return from `/orders/:orderNumber` within a return window.
- Admin: approve/reject, trigger Razorpay refund on "received".
- Reuse the existing `order_events` timeline for return milestones.

### 2.2 Discounts / coupons
- Schema: `coupons` (code, type `percent|flat`, value, min_cart, usage_limit, per_user_limit, starts_at, ends_at, active).
- Checkout: apply/validate a code, recompute totals server-side (never trust the client). Extend `lib/pricing.ts`.
- Admin: CRUD + usage analytics. Audit-log every coupon mutation.
- Contracts: bump (minor) for the new cart/checkout fields.

### 2.3 Wishlists
- Schema: `wishlist_items` (user_id, sku_id, created_at), unique per (user, sku).
- Endpoints: `GET/POST/DELETE /me/wishlist`. Guest wishlist merges on login (same pattern as guest cart).
- Lightweight; good first v1.5 task.

### 2.4 Abandoned-cart recovery
- Inngest scheduled function: find carts with items + no order after N hours, owner has an email/phone.
- Send one reminder (Resend email; optionally MSG91), throttled, with an unsubscribe/opt-out.
- Respect consent — do not message guests who didn't opt in.

## 3. Tech-debt / hardening (opportunistic)

- Refresh-token **rotation reuse detection**: v1 revokes jti on logout (`lib/token-revocation.ts`). v1.5 could also revoke the old jti on every rotation and treat reuse of a rotated token as a breach signal (force-logout all sessions).
- Move the in-memory fallbacks (`cache.ts`, `idempotency.ts`, `ratelimit.ts`, `token-revocation.ts`) to Redis-only in prod once multi-instance, and assert Upstash is configured at boot.
- Backfill load/perf tests for `/search` and `/category/[slug]` under cache miss.

## How to graduate an item from this doc

1. Confirm v1 is stable and the item is the highest-value next thing (check Sentry/Axiom + customer feedback, not just this list).
2. Write a phase-style doc (`docs/phases/phase-7-*.md`) with tasks, acceptance criteria, and a contracts version plan.
3. Add an integration milestone if it touches the FE contract.
4. Move the item from this backlog into that phase doc.
